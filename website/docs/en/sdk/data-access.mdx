# Programmatic data access

`@vitest-agent/sdk` exposes the full data layer through two Effect services — `DataStore` for writes and `DataReader` for reads — each with a live layer for production use and test layers for isolation. A CLI escape hatch lets you run ad-hoc SQL against the database without writing any Effect code.

## DataStore and DataReader

Both services are Effect `Context.Tag` definitions. You request them through `Effect.flatMap` or `Effect.gen` just like any other Effect service.

```ts
import { Effect } from "effect";
import { DataReader, DataStore } from "@vitest-agent/sdk";

// Read the latest run for a project (resolves to Option<AgentReport>)
const readLatest = Effect.flatMap(DataReader, (reader) =>
  reader.getLatestRun("my-project")
);

// Write a note; resolves to the new note's row id
const writeNote = Effect.flatMap(DataStore, (store) =>
  store.writeNote({
    title: "Observation",
    content: "The login flow is consistently slow under load.",
    scope: "global",
  })
);
```

`DataStore` owns all SQLite writes. `DataReader` owns all reads and assembles domain types from raw row data via the assemblers in `sql/assemblers.ts`. Neither service opens a database connection directly — they both depend on an underlying `SqlClient` provided by the layer you compose.

### Live layers

The live layers wire the services to a real SQLite file. You compose them with a `SqliteClient` layer from `@effect/sql-sqlite-node`:

```ts
import { Effect, Layer } from "effect";
import { DataReader, DataReaderLive, DataStore, DataStoreLive } from "@vitest-agent/sdk";
import { SqliteClient } from "@effect/sql-sqlite-node";
import { NodeContext } from "@effect/platform-node";
import { resolveDataPath } from "@vitest-agent/sdk";

// Resolve the path via the five-source XDG chain
const dbPath = "/path/to/data.db"; // or use resolveDataPath(cwd) in an Effect

const SqliteLayer = SqliteClient.layer({ filename: dbPath });

const AppLayer = Layer.mergeAll(DataReaderLive, DataStoreLive).pipe(
  Layer.provide(SqliteLayer),
  Layer.provide(NodeContext.layer)
);

await Effect.runPromise(
  Effect.provide(readLatest, AppLayer)
);
```

:::tip
Use `resolveDataPath(cwd)` from `@vitest-agent/sdk` to derive the correct `dbPath` from your workspace identity instead of hard-coding a path. It returns an `Effect<string, WorkspaceRootNotFoundError>`.
:::

### Error handling

`DataStore` write methods wrap their SQL in `Effect.try` and surface failures as `DataStoreError` tagged with `operation`, `table`, and a `reason` extracted from the underlying SQLite message. The plugin logs errors to stderr and never crashes the test run on a DataStore failure.

`DataReader` read methods return empty records when queried data does not exist rather than failing.

## Testing utilities

The `@vitest-agent/sdk/testing` sub-path exports two primitives and five preset factory functions for seeding test state without writing fixtures manually.

### makeTestLayer

`makeTestLayer(filename)` creates an Effect `Layer` that provides both `DataStore` and `DataReader` backed by a fresh in-memory (or file-backed) SQLite database, with migrations already applied:

```ts
import { Effect, Option } from "effect";
import { DataReader } from "@vitest-agent/sdk";
import { makeTestLayer } from "@vitest-agent/sdk/testing";

const layer = makeTestLayer(":memory:");

it("returns none when no runs exist", async () => {
  const result = await Effect.runPromise(
    Effect.provide(
      Effect.flatMap(DataReader, (r) => r.getLatestRun("default")),
      layer
    )
  );
  expect(Option.isNone(result)).toBe(true);
});
```

Pass `":memory:"` for full test isolation. Pass a file path for tests that need to inspect the on-disk state.

### DataStoreTestLayer

`DataStoreTestLayer` is a convenience shorthand for `makeTestLayer(":memory:")`. Use it when you only need the default in-memory configuration and do not want to call the function explicitly:

```ts
import { DataStoreTestLayer } from "@vitest-agent/sdk/testing";

// Equivalent to makeTestLayer(":memory:")
const layer = DataStoreTestLayer;
```

### Preset factories

Five preset factories seed representative database states. Each factory is a function that accepts a `dbPath` (pass `":memory:"`) and returns an Effect `Layer`:

| Factory | What it seeds |
|---------|---------------|
| `empty` | A migrated database with no rows |
| `singlePassingRun` | One run record with all test cases passing |
| `withFailures` | One run record with some test cases failing |
| `flaky` | Multiple runs with a flaky test pattern (some passes, some failures) |
| `withTddTask` | A seeded TDD task with goals and behaviors |

```ts
import { Effect } from "effect";
import { DataReader } from "@vitest-agent/sdk";
import {
  empty,
  singlePassingRun,
  withFailures,
  flaky,
  withTddTask,
} from "@vitest-agent/sdk/testing";

describe("classification", () => {
  it("labels a consistently failing test as persistent", async () => {
    const layer = withFailures(":memory:");
    const result = await Effect.runPromise(
      Effect.provide(
        Effect.flatMap(DataReader, (r) => r.getLatestRun("default")),
        layer
      )
    );
    // result is Option<AgentReport>; unwrap it and assert on the report summary
  });
});
```

Each factory returns a `Layer` you provide directly in your test. There is no global setup or teardown — each test gets a fresh isolated database.

## Ad-hoc SQL via the CLI

For one-off queries during debugging or exploration, the `db query` command gives you a read-only SQL prompt against the live `data.db`:

```bash
vitest-agent db query "SELECT project, count(*) FROM test_runs GROUP BY project"
```

The command accepts a SQL string as an argument. All queries run in read-only mode — writes are rejected at the connection level so you cannot accidentally corrupt production data. Output is formatted as a table by default; pass `--format json` for machine-readable output.

See [CLI commands](/cli/commands) for the full `db` command reference including `db path`, `db prune`, and `db reset`.

## Effect-idiomatic patterns

### Composing reads and writes

Effects compose naturally. A common pattern in tests is to write data with `DataStore`, then read it back with `DataReader`, all within a single `Effect.gen`:

```ts
import { Effect } from "effect";
import { DataStore, DataReader } from "@vitest-agent/sdk";
import { makeTestLayer } from "@vitest-agent/sdk/testing";

const program = Effect.gen(function* () {
  const store = yield* DataStore;
  const reader = yield* DataReader;

  yield* store.writeNote({
    title: "Coverage dropped",
    content: "Branches fell below 65% after the refactor.",
    scope: "project",
    project: "@vitest-agent/sdk",
  });

  const notes = yield* reader.getNotes("project", "@vitest-agent/sdk");
  return notes;
});

const result = await Effect.runPromise(
  Effect.provide(program, makeTestLayer(":memory:"))
);
```

### Providing layers in production

Outside test code, compose your layer once and reuse it across all program entry points:

```ts
import { Layer, ManagedRuntime } from "effect";
import { DataReaderLive, DataStoreLive, resolveDataPath } from "@vitest-agent/sdk";
import { SqliteClient } from "@effect/sql-sqlite-node";
import { NodeContext, NodeFileSystem } from "@effect/platform-node";
import { Effect } from "effect";

const buildLayer = (cwd: string) =>
  Effect.gen(function* () {
    const dbPath = yield* resolveDataPath(cwd);
    const SqliteLayer = SqliteClient.layer({ filename: dbPath });
    return Layer.mergeAll(DataReaderLive, DataStoreLive).pipe(
      Layer.provide(SqliteLayer),
      Layer.provide(NodeContext.layer),
      Layer.provide(NodeFileSystem.layer)
    );
  });
```

`resolveDataPath` requires `NodeFileSystem.layer` (or equivalent) in its environment because it reads `package.json` and optional TOML config from disk.

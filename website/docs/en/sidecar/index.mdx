# @vitest-agent/sidecar

A Node Single Executable Application (SEA) binary for the per-Bash-call `inject-env` hot path. Every Bash tool call from a Claude Code agent fires the plugin's `pre-tool-use/bash.sh` hook, which detects Vitest invocations and prepends the canonical `VITEST_AGENT_*` env prefix. Running that through the full `vitest-agent` CLI pays Node cold-start latency on every call; this binary runs the same logic with a fraction of the startup cost.

This package arrives automatically via `@vitest-agent/cli`, which is itself a required peer of `@vitest-agent/plugin`. You rarely need to add it directly.

## Features

- **Native binary** â€” Node SEA wrapping the same `inject-env` TypeScript that ships in `@vitest-agent/cli`; the JS fallback and binary stay byte-identical in behavior
- **Per-platform sub-packages** â€” `@vitest-agent/sidecar-darwin-arm64`, `@vitest-agent/sidecar-linux-arm64`, `@vitest-agent/sidecar-linux-x64`, `@vitest-agent/sidecar-win32-x64` as `optionalDependencies`; only the matching platform installs
- **`resolveSidecarBinaryPath()`** â€” exported function that returns the absolute path of the installed binary, or `null` on unsupported platforms

## Install

The recommended install path is through the plugin:

```bash
npm install --save-dev @vitest-agent/plugin
# @vitest-agent/sidecar arrives transitively through @vitest-agent/cli
```

To install standalone:

```bash
npm install --save-dev @vitest-agent/sidecar
```

pnpm installs only the matching per-platform child (`optionalDependencies` filtered by `os`/`cpu` fields). On an unsupported platform the parent package installs but no binary is present; the hook falls back to the JS CLI automatically.

## How the binary path reaches the hook

The plugin's hooks use the binary via a two-step protocol:

1. At session start, the `SessionStart` hook runs `vitest-agent agent sidecar-path` once and exports the result as `VITEST_AGENT_SIDECAR_BIN`.
2. On every subsequent Bash tool call, the `PreToolUse` hook reads `$VITEST_AGENT_SIDECAR_BIN` and execs the binary directly â€” no PATH lookup, no Node startup.

If `VITEST_AGENT_SIDECAR_BIN` is unset (unsupported platform, fresh session before the env var propagates), the hook falls back to `vitest-agent inject-env` via the JS CLI.

## `resolveSidecarBinaryPath`

The exported function is used by `vitest-agent agent sidecar-path`:

```ts
import { resolveSidecarBinaryPath } from "@vitest-agent/sidecar";

const binaryPath = resolveSidecarBinaryPath();
// Returns an absolute path string, or null if the platform binary is not installed
```

The resolver uses `require.resolve` anchored inside `@vitest-agent/sidecar`'s own `package.json` to find the `optionalDependency` binary. This is why the resolver cannot be moved to another package â€” `require.resolve` only locates `optionalDependencies` declared in the package that calls it.

## Per-platform packages

The four child packages each declare their own `bin` entry pointing at the prebuilt SEA binary:

| Package | Platform |
|---------|----------|
| `@vitest-agent/sidecar-darwin-arm64` | macOS Apple Silicon |
| `@vitest-agent/sidecar-linux-arm64` | Linux ARM64 |
| `@vitest-agent/sidecar-linux-x64` | Linux x64 |
| `@vitest-agent/sidecar-win32-x64` | Windows x64 |

Each child's binary bundles `dispatch` from `@vitest-agent/sdk/dispatch` â€” the minimal entry point that carries no Effect or SQLite data layer â€” keeping the SEA bundle small.
